実践編 ・ 柱① AI基盤を選ぶ

AIの「足腰」を、
3つの基盤で整える。

エージェント時代の主役は、目立つ機能より「土台」。トークン・環境・コード理解の3つを押さえれば、上に乗るAIが一段速く・賢くなります。

headroom ECC codegraph

この3つは競合ではなく、役割の異なる土台です

なぜ「基盤」なのか

高機能なAIだけでは、足りない

基盤がないと
  • 毎回大量の文脈を投げ、トークンを浪費する
  • 環境設定がバラバラで、エージェントの挙動が安定しない
  • コードを毎回探索させ、応答が遅くブレる
  • スキル・記憶・権限が個人に閉じてしまう
基盤を整えると
  • 渡す前に圧縮して、コストと速度を両立できる
  • 環境・スキル・記憶がパックで揃い、再現性が出る
  • コード構造を先に持たせ、ピンポイントで答えさせる
  • チームで同じ土台を共有でき、属人化が解ける
基盤の三役

基盤 ①

headroom ─ 渡す前に、軽くする

1
headroom
INPUT COMPRESSION
ツール出力・ログ・ファイル・RAGの塊を、AIに渡す前に圧縮。応答品質を保ったまま、入力トークンだけを大幅に減らす。
向いている使い方
  • 長いログや検索結果をエージェントに渡す前段に挟む
  • RAGで取り出した文書束のサイズを抑える
  • 運用コストの上振れを止めたい本番環境
圧縮率の目安
60 〜 95%
回答は変えずに、入力だけを削れる範囲。
提供形態
ライブラリ／プロキシ／MCPサーバーの3形態。既存パイプに差し込みやすい。
ひと言で
「AIに渡す前のフィルター」。賢くする道具ではなく、賢さを保ったまま軽くする道具。
基盤 ②

ECC ─ エージェント環境を、まとめて底上げ

2
ECC
AGENT ENVIRONMENT PACK
Claude Code・Cursor・Codex などのエージェント環境を、スキル・記憶・セキュリティを1パックで揃えて強化するブースト集。
向いている使い方
  • チーム全員の開発エージェントを同じ土台に揃える
  • スキルや記憶を個人運用から組織運用へ移す
  • セキュリティ要件を満たした状態で配布する
同梱されるもの
64 × 261
64エージェント、261スキル。すぐ動く形でまとまっている。
対応する主な環境
Claude Code／Cursor／Codex などのコーディング系エージェントをまたいで適用。
ひと言で
「個人技を、組織の標準装備に」。属人化していたスキルを共有資産に変える。
基盤 ③

codegraph ─ コードに、先に地図を持たせる

3
codegraph
CODE KNOWLEDGE GRAPH
コードベースをあらかじめナレッジグラフ化。AIが毎回ファイルを探し回らず、関係する箇所へ最短で到達する。完全ローカルで動く。
向いている使い方
  • 大規模リポジトリでのコード理解・改修
  • 外部にコードを出せないクローズド環境
  • ツール呼び出しの暴発を抑えたいエージェント運用
ベンチマーク平均
−47% / −58%
トークン消費と、ツール呼び出し回数。どちらも大きく下がる。
動作環境
完全ローカル動作。コードを外に出さずに、構造理解を持たせられる。
ひと言で
「探さずに、知っている状態」。AIに地図を渡すことで、毎回の探索を省く。
